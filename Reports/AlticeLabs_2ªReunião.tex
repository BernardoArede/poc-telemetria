\section{Estado da Arte}
    \subsection{Desafio em arquiteturas distribuídas}
        Nas últimas décadas, o paradigma de desenvolvimento de software passou de arquiteturas monolíticas para ecossistemas baseados em microserviços e aplicações cloud-native. Com o crescimento destas arquiteturas distribuídas, a observabilidade tornou-se um fator crítico para garantir não só o desempenho e a fiabilidade, mas também a rápida deteção e correção de falhas.
        
        Num sistema monolítico, o fluxo de execução é linear e ocorre num único domínio, o que facilita o isolamento de erros. Contudo, em ambientes modernos, um único pedido de um utilizador pode atravessar dezenas de microserviços diferentes, desenvolvidos em linguagens diferentes e alojados em múltiplas arquiteturas. Neste contexto, as ferramentas tradicionais de monitorização revelam imensas limitações quando aplicadas a sistemas complexos e altamente distribuídos. A monitorização clássica, focada apenas na verificação de estados de alerta, tornou-se insuficiente para responder à verdadeira questão no contexto de sistemas distribuídos: ``Porquê e onde é que o sistema está a falhar?''.

    \subsection{Paradigma da Observabilidade}
        Para dar resposta a esta complexidade, a indústria adotou o conceito de Observabilidade. Ao contrário da monitorização tradicional, que é reativa, a observabilidade é uma propriedade do sistema que permite inferir o seu estado interno a partir das suas saídas externas (telemetria). Este conceito apoia-se convencionalmente em três pilares fundamentais de dados:

        \begin{itemize}
        \item \textbf{Métricas (Metrics):} Representações numéricas de dados agregados ao longo do tempo (ex.: taxa de uso de CPU, número de pedidos por segundo), são cruciais para a deteção rápida de anomalias do sistema.
        
        \item \textbf{Traces:} Representam o ciclo de vida completo de um pedido à medida que este viaja pelos vários componentes de um sistema distribuído. Permitem identificar problemas de performance e mapear dependências entre serviços \cite{jaeger_docs}.
        
        \item \textbf{Logs:} Registos imutáveis e discretos de eventos que ocorreram no sistema. Oferecem o contexto de mais alto nível de detalhe (\textit{debugging}) quando um erro específico é detetado \cite{otel_video}.
        \end{itemize}

    \subsection{OpenTelemetry}
        Historicamente, a recolha de sinais de telemetria sofria de uma forte dependência de fornecedores. Cada ferramenta de monitorização exigia a instalação de agentes proprietários ou a integração de bibliotecas específicas no código-fonte, o que elevava o custo técnico e limitava a flexibilidade.

        Para combater este problema, o OpenTelemetry surge como uma solução unificada e independente de fornecedor, sendo hoje amplamente adotada pela indústria. Trata-se de um projeto da Cloud Native Computing Foundation (CNCF) que fornece um conjunto de APIs, SDKs e ferramentas padrão para a instrumentação, recolha e exportação de dados de telemetria \cite{otel_docs}. Ao padronizar a instrumentação e permitir a integração com diversas plataformas de monitorização, o OpenTelemetry separa a recolha dos dados da sua camada de armazenamento e visualização. Isto contribui diretamente para a redução da complexidade técnica e para a melhoria da eficiência operacional das equipas.

    \subsection{O Ecossistema de Análise e Visualização}
    A arquitetura proposta neste estágio integra o OpenTelemetry com algumas das ferramentas open-source mais consolidadas no mercado, garantindo um pipeline de observabilidade robusto:

\begin{itemize}
    \item  \textbf{Prometheus:} O padrão de facto para o armazenamento em séries temporais e alerta de métricas em ambientes cloud-native.
    \item \textbf{Jaeger:} Plataforma especializada na receção, armazenamento e visualização avançada de rastreios distribuídos \cite{jaeger_docs}.
    \item \textbf{Grafana e Loki:} O Grafana atua como a interface unificada para a correlação visual de todos os sinais, enquanto o Loki é otimizado para a agregação de logs com alta eficiência, partilhando sinergias com o modelo de labels do Prometheus.
\end{itemize}
    A interligação destas tecnologias com serviços desenvolvidos em linguagens como Golang, orquestrada pelo OpenTelemetry Collector, forma a base tecnológica que sustenta os objetivos práticos deste projeto.

\section{Zero-code Implementation e Auto-instrumentação}
    \subsection{O Paradigma Low-code na Observabilidade}
        Em alinhamento com os objetivos propostos para este estágio, explorou-se a adoção de abordagens \textit{low-code} e \textit{zero-code} na implementação da observabilidade. Tradicionalmente, a instrumentação de aplicações exigia alterações significativas no código-fonte para injetar chamadas a bibliotecas de monitorização, um processo moroso e com elevado risco de introdução de erros em ambientes de produção.
        
        A abordagem \textit{zero-code} elimina esta barreira. Utilizando agentes dinâmicos ou tecnologias ao nível do kernel, torna-se possível extrair telemetria rica (métricas e traces) de uma aplicação sem alterar uma única linha do seu código original.

    \subsection{Mecanismos de Zero-Code no OpenTelemetry}
        O ecossistema OpenTelemetry suporta nativamente a auto-instrumentação para diversas linguagens de programação, incluindo Golang (uma das tecnologias a utilizar neste projeto) \cite{otel_go_zerocode}. Este mecanismo funciona através das bibliotecas padrão e \textit{frameworks} mais populares (como HTTP clients, drivers de base de dados e servidores web) \cite{otel_go_instr}.
        
        Ao iniciar a aplicação com o agente do OpenTelemetry acoplado, este interceta automaticamente as requisições, gerando os \textit{spans} correspondentes, propagando o contexto distribuído e exportando os dados no formato OTLP para o Collector.

    \subsection{Vantagens para o Ecossistema Altice Labs}
        A validação desta prova de conceito tem um impacto direto no contexto empresarial da Altice Labs. A capacidade de implementar observabilidade \textit{zero-code} oferece vantagens estratégicas e operacionais consideráveis:
        \begin{itemize}
            \item \textbf{Adoção sem atrito:} Equipas com serviços atribuídos (\textit{legacy}) podem obter visibilidade imediata sem necessitarem de alocar tempo de desenvolvimento para refatorização de código.
            \item \textbf{Padronização imediata:} Garante que todos os serviços exportam telemetria com as mesmas \textit{Semantic Conventions} do OpenTelemetry \cite{otel_docs}, facilitando a correlação no Grafana.
            \item \textbf{Redução do débito técnico:} Separa a lógica de negócio da lógica de monitorização, mantendo os repositórios de código mais limpos e fáceis de manter.
        \end{itemize}

\section{Arquitetura da Prova de Conceito}
    \subsection{Visão Geral do Sistema}
        A definição da arquitetura do protótipo visa estabelecer um fluxo de dados de telemetria fiável, escalável e agnóstico em relação à infraestrutura subjacente. O desenho da solução assenta num modelo produtor-consumidor, onde os serviços geram sinais de observabilidade que são centralizados, processados e posteriormente encaminhados para as respetivas bases de dados especializadas (\textit{backends}).

    \subsection{Componentes e Tecnologias}
        Para cumprir os requisitos de integração e padronização, a infraestrutura foi dividida em três camadas lógicas principais, suportadas por um conjunto de tecnologias \textit{open-source}:
        \begin{itemize}
            \item \textbf{Camada de Aplicação (Geração):} Dois microserviços desenvolvidos em \textbf{Golang}  que simulam transações de negócio. Estes serviços utilizam os SDKs do OpenTelemetry para instrumentar o código e exportar métricas, logs estruturados e \textit{traces}.
            \item \textbf{Camada de Processamento (Pipeline):} Materializada pelo \textbf{OpenTelemetry Collector}, que atua como o \textit{gateway} único de telemetria da rede.
            \item \textbf{Camada de Armazenamento e Visualização (\textit{Backends}):}
            \begin{itemize}
                \item \textbf{Prometheus:} Receção e armazenamento em séries temporais das métricas de performance.
                \item \textbf{Jaeger:} Armazenamento e indexação dos \textit{spans} para visualização do rastreio distribuído.
                \item \textbf{Grafana Loki:} Agregação de \textit{logs} otimizada, permitindo a correlação direta através de \textit{trace\_ids}.
                \item \textbf{Grafana:} Plataforma de interface de utilizador (UI) para construção de \textit{dashboards} unificados.
            \end{itemize}
        \end{itemize}

    \subsection{O Papel Central do OpenTelemetry Collector}
        A decisão arquitetural de não enviar a telemetria diretamente das aplicações para os \textit{backends} justifica-se pelos princípios de boas práticas em sistemas distribuídos. O OpenTelemetry Collector atua como um \textit{proxy} independente, estruturado num \textit{pipeline} de três fases:
        \begin{enumerate}
            \item \textbf{Receivers:} Configurados para aceitar dados via protocolo \textbf{OTLP} (OpenTelemetry Line Protocol) via gRPC e HTTP, garantindo uma interface de entrada padronizada.
            \item \textbf{Processors:} Responsáveis pela manipulação dos dados em trânsito. Permitem o processamento em \textit{batch} (para otimizar o uso da rede), a adição de metadados globais (ex.: nome do ambiente ou \textit{datacenter}) e a filtragem de dados sensíveis.
            \item \textbf{Exporters:} Desacoplam as aplicações do destino final dos dados. O Collector foi configurado com exportadores nativos para encaminhar as métricas para o Prometheus, os \textit{traces} para o Jaeger e os \textit{logs} para o Loki, traduzindo o formato OTLP para as APIs específicas de cada ferramenta, se necessário.
        \end{enumerate}

    \subsection{Justificação da Escolha do Collector}
        A decisão de introduzir o OpenTelemetry Collector como peça central da arquitetura baseia-se na necessidade de criar um intermediário agnóstico ao fornecedor. O Collector atua como o motor de receção (via protocolo OTLP), processamento (agregação, filtragem) e exportação dos dados, aliviando os microserviços dessa carga de processamento e garantindo maior segurança e flexibilidade no encaminhamento da telemetria.
